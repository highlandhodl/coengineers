## How to Use This Backlog

1. **List all potential work** - features, stories, bugs, technical tasks
2. **Prioritise ruthlessly** - most important items at top
3. **Size each item** - Small, Medium, Large (time/complexity estimate)
4. **Mark status** - Not Started, In Progress, Done
5. **Review regularly** - update priorities as you learn

---

## Priority Levels

- **P0 - Critical:** Must have for v1, blocking launch
- **P1 - High:** Important for v1, ship if possible
- **P2 - Medium:** Nice to have for v1, or planned for v2
- **P3 - Low:** Backlog items for future consideration

## Definition of Ready

Before moving a story to "In Progress", ensure:

- [ ] Story is clearly defined
- [ ] Acceptance criteria are written
- [ ] Technical approach is understood
- [ ] Dependencies are identified
- [ ] Size estimate is agreed

---

## Definition of Done

Before marking a story "Done", ensure:

- [ ] Functionality works as specified
- [ ] Code is tested (manual or automated)
- [ ] Documentation is updated
- [ ] Acceptance criteria are met
- [ ] No critical bugs
